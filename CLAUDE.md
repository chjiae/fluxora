# Fluxora — Claude Code 项目指令

本文件位于仓库根目录，供 Claude Code 在本项目中自动加载。开始任何分析、设计、编码、数据库迁移、测试、审阅或 Git 操作前，必须先完整阅读根目录的 [AGENT.md](./AGENT.md)。

`AGENT.md` 是本仓库唯一、完整且权威的工程规则来源；本文件是面向 Claude Code 的入口与执行摘要，不得与其冲突或弱化其中任何要求。若两者存在理解差异，以 `AGENT.md` 为准。

## 强制规则

### 代码、数据库与性能

- 实体类、实体字段、关键代码、复杂逻辑和易错逻辑必须添加有实际业务意义的中文注释；说明意图、约束、边界、状态流转或安全原因，禁止机械复述代码。
- 所有 Flyway 迁移脚本必须包含清晰的中文业务注释。PostgreSQL 业务表还必须在 DDL 中使用 `COMMENT ON TABLE` 和 `COMMENT ON COLUMN` 注释表及关键字段，特别是作用域、逻辑删除、过期时间、密码哈希、角色/权限关联、审计字段和唯一约束。
- 所有数据库 SQL 只能写入 MyBatis XML；禁止 Java 注解 SQL。
- 不得在循环中逐项调用 HTTP 接口、Mapper 或单条 SQL，禁止 N+1 查询/写入。涉及多个对象时优先设计集合化接口，并使用批量查询、`IN`、关联查询、批量 `INSERT`/`UPDATE`/`DELETE` 和合理事务边界。
- 只有存在严格顺序依赖且无法批量处理时才允许逐项操作；必须在关键代码处用中文说明业务原因、数据量边界和性能影响。
- 新增接口、Mapper XML 和服务逻辑必须考虑分页、索引命中、批量参数规模、事务范围与避免重复查询。

### 软删除

- 业务表的逻辑删除统一使用 `deleted_at TIMESTAMPTZ NULL`，禁止使用 `is_deleted BOOLEAN` 等布尔标记。`NULL` 表示未删除；删除时执行 `UPDATE … SET deleted_at = NOW(), updated_at = NOW() WHERE deleted_at IS NULL`，不得物理删除核心业务数据。
- 删除后允许复用的业务唯一字段（如租户码、用户名、邮箱）必须使用 `WHERE deleted_at IS NULL` 部分唯一索引；不允许复用时在迁移注释中说明原因。
- 所有业务查询、列表、详情、计数、唯一性检查的 SQL 默认追加 `AND deleted_at IS NULL`；仅在认证、审计、恢复等明确场景下提供命名带 `IncludeDeleted` 的独立方法，并附中文注释说明用途与调用方。
- JWT、Session、登录与状态过滤器在每次请求时必须校验目标账号 / 租户 `deleted_at IS NULL`；软删除后已签发的凭证必须立即失效。
- 实体字段命名为 `deletedAt`（`Instant`），三态状态（ENABLED / DISABLED / DELETED）通过 `getStatus()` 在 Java 端派生，不得入库。响应 DTO 不向普通用户返回 `deletedAt`，仅返回派生 `status`。
- 软删除字段的写入与恢复只能由服务层统一管理；Controller、Job、外部脚本不得直接更新 `deleted_at`。仓库内仍在使用 `is_deleted BOOLEAN` 的历史表必须在最近的 Flyway 迁移中改造为 `deleted_at`，禁止新增任何 `is_deleted BOOLEAN` 字段。

### 安全错误提示与交互

- 普通用户可见页面、弹窗、Toast、表单提示和错误页不得显示 HTTP 状态码、业务错误码、异常类名、堆栈、SQL/数据库错误、字段名、后端原始异常、内部路径、配置、Token、密码或其他敏感信息。
- 后端仍返回正确 HTTP 状态码给程序使用；返回结构必须包含稳定业务错误码、用户可理解的中文提示和可选追踪标识，但不得泄露内部实现细节或敏感信息。
- 前端不得直接展示后端原始 `message`、HTTP 状态码、业务错误码或异常文本；必须通过统一错误映射输出安全中文文案。
- 登录失败固定提示“用户名或密码错误，请重新输入”；登录失效提示“登录已失效，请重新登录”；无权限提示“当前账号没有此操作权限”。
- 表单校验优先贴近字段显示明确修改提示；非表单失败使用统一反馈组件；提交失败必须保留已填写内容。
- 生产环境不得向 UI 暴露调试信息。新增页面与接口必须复用统一错误处理机制。

### 验证与 Git

- 变更前先检查现有代码、模块边界与 `git status`；不要覆盖用户已有变更。
- 实现后运行与修改匹配的测试、构建和必要的 Playwright 验证。错误密码、无权限、字段校验、网络失败、超时、服务异常等场景必须验证 UI 不暴露技术文本。
- 使用 Conventional Commits：`feat:`、`fix:`、`docs:`、`refactor:`、`build:`、`chore:` 等，描述使用中文。
- 提交前执行 `git status` 与 `git diff --check`；优先 `git add <具体文件>`，禁止盲目 `git add .`。
- 不提交密钥、密码、Token、真实环境变量、数据库数据或 IDE 临时文件。

## 工作方式

1. 先阅读 `AGENT.md`、相关模块代码、配置和测试，再给出或执行方案。
2. 不为方便实现而越过模块边界、弱化安全规则或提前创建无关领域模型。
3. 需求与上述规则冲突、规则不明确，或实现必然违反规则时，先说明冲突并请求澄清；不得自行降低标准。
4. 对框架、库、SDK、CLI 或云服务的当前 API/配置不确定时，优先使用 Context7 或官方文档验证，禁止凭记忆猜测。
