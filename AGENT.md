# AGENT.md

## 中文注释与代码规范

所有实体类、实体字段、关键代码、复杂逻辑和容易踩坑的逻辑必须添加中文注释。

注释应说明业务意图、约束、边界、状态流转原因或安全考虑，不得机械重复代码的字面含义，也不要为明显的样板代码堆砌无意义注释。

所有 Flyway 迁移脚本必须包含清晰且有实际意义的中文注释，说明表、字段、约束、索引和初始化数据的业务目的。

对于 PostgreSQL 业务表，除 SQL 文件内注释外，还必须使用 `COMMENT ON TABLE` 和 `COMMENT ON COLUMN` 将中文业务说明写入数据库 DDL 元数据。每张业务表及其关键字段都必须覆盖，尤其是作用域、逻辑删除、过期时间、密码哈希、角色/权限关联、审计字段和唯一约束相关字段；不得仅保留 SQL 文件中的行注释。

所有数据库 SQL 必须写在 MyBatis XML 文件中，禁止使用 Java 注解编写 SQL。

## 性能与批量数据访问规范

不得为了实现方便在循环中逐项调用 HTTP 接口、Mapper 方法或单条 SQL，从而形成循环调用接口、循环执行数据库操作或 N+1 查询/写入问题。

涉及多个对象的查询、权限加载、关联数据加载、状态变更、关联关系维护或逻辑删除时，应优先设计集合化接口，并使用批量查询、`IN` 条件、关联查询、批量 `INSERT`、批量 `UPDATE` 或批量 `DELETE` 等方式，在合理的事务边界内完成。

禁止以下实现方式：先查询列表，再对列表中的每一项分别请求详情、查询关联数据或执行一次数据库更新；也不得由前端循环调用单对象接口来完成批量业务操作。

仅当操作之间确实存在严格顺序依赖、单项结果决定后续处理，或数据量已被明确限制且无法使用集合化语义时，才允许逐项处理。此类例外必须在关键代码处添加中文注释，说明无法批量处理的业务原因、数据量边界和性能影响。

新增接口、Mapper XML 和服务逻辑必须考虑分页、索引命中、批量参数规模、事务范围与避免重复查询；不得以牺牲数据库性能换取实现上的便利。

## 软删除字段规范

业务表的逻辑删除统一使用 `deleted_at TIMESTAMPTZ NULL` 时间戳字段，禁止使用 `is_deleted BOOLEAN` 等布尔标记。

### 字段与索引

1. 所有支持逻辑删除的业务表必须包含 `deleted_at TIMESTAMPTZ` 列，默认值为 `NULL`，并通过 `COMMENT ON COLUMN` 写明：`NULL 表示未删除；非 NULL 表示已删除并记录删除时刻，用于审计、未来窗口期恢复与排查`。
2. 删除操作执行 `UPDATE … SET deleted_at = NOW(), updated_at = NOW() WHERE deleted_at IS NULL`，禁止物理删除核心业务数据；重复删除已删行的请求应返回成功幂等响应或忽略，不得报错。
3. 删除后允许复用的业务唯一字段（如租户码、用户名、邮箱）必须使用部分唯一索引：`CREATE UNIQUE INDEX … ON table (field) WHERE deleted_at IS NULL;`。删除后不允许复用的字段也应在文件头注释处明确说明原因，避免无意中阻断未来的重用需求。
4. 高频查询路径上的 `deleted_at IS NULL` 过滤条件如果与租户、状态等组合，应通过额外索引覆盖；不得依赖全表扫描后过滤。

### 查询与服务层

1. 所有业务查询、列表、详情、计数、唯一性检查的 SQL 默认追加 `AND deleted_at IS NULL` 过滤；不得返回已删除记录。
2. 仅在认证阶段、审计接口、恢复流程等明确需要识别已删除记录的场景，提供独立的 `findByIdIncludeDeleted` 类方法，命名必须显式包含 `IncludeDeleted` 或 `Deleted`，且在 XML/Java 中均附中文注释说明用途与调用方。
3. JWT、Session、登录、状态过滤器等长会话路径必须在每次请求时校验目标账号 / 租户 `deleted_at IS NULL`；软删后已签发的凭证必须立即失效。
4. 软删除字段的写入只能由服务层统一管理（如 `MemberService.softDelete`、`TenantService.softDelete`），禁止在 Controller、Job、外部脚本中直接更新 `deleted_at`。
5. 对软删除字段的反操作（恢复）必须经过显式的服务方法和权限校验，禁止前端、调用方直接置 `deleted_at = NULL`。

### 实体层

1. 实体类对应字段命名为 `deletedAt`，类型 `Instant`，附中文注释说明业务语义。
2. 实体如需对外暴露三态状态（如 ENABLED / DISABLED / DELETED），通过 `getStatus()` 类计算方法在 Java 端派生，禁止把状态枚举写入数据库列。
3. 响应 DTO 不得直接返回 `deletedAt` 时间戳给普通用户，必要时仅返回派生 `status` 字段。

### 历史遗留

仓库内任何仍在使用 `is_deleted BOOLEAN` 的表都视为历史遗留，必须在最近的 Flyway 迁移中改造为 `deleted_at TIMESTAMPTZ NULL`，同步迁移历史数据（`is_deleted = true` 的行将 `deleted_at` 置为 `updated_at`），并删除旧的 `is_deleted` 列与对应布尔索引。不允许新增任何使用 `is_deleted BOOLEAN` 的表或列。

## 控制台页面设计规范

控制台所有管理类页面（租户管理、成员管理及后续模块）必须遵循统一的「克制、可靠、敏锐」产品调性，禁止套用模板化 SaaS 后台样式。

### 页面骨架与节奏

每个管理页采用四段式 CSS Grid：页头 / 指标条 / 工具栏 / 表格区(1fr) / 分页。`grid-template-rows: auto auto auto 1fr auto`，`gap: 20px`。仅表格区内部滚动，其他三行常驻可见。新增任何一个一级管理页面，都必须保留这个节奏，不允许把表格直接顶到页头下方造成「臃肿、无留白」的感觉。

总览页采用 `display: flex; flex-direction: column; gap: 24px` 的更宽节奏：标题段 → 指标条 → 内容分区。

### 指标条（MetricStrip）

任何「列表 + 操作」型管理页都必须在页头与工具栏之间提供聚合指标条；总览页同样使用。指标条用于给用户立即建立上下文（多少租户、多少启用、多少快过期、多少角色分布等），减少先扫表格再心算的负担。

强制要求：

1. 复用 `fluxora-web/src/components/MetricStrip.vue`，不允许在页面内单独发明指标视觉语言。
2. 指标条数据必须来自后端**单次聚合 SQL**（如 `/api/tenant/stats`、`/api/members/stats`），禁止前端遍历列表自行统计或并发拉多个计数接口。聚合 SQL 必须写在对应 Mapper XML 中，统一使用 PostgreSQL `COUNT(*) FILTER (WHERE …)` 表达式，保持 `deleted_at IS NULL` 过滤。
3. 指标条只展示「计数」类标量，每个数值伴随：标签（标签在上）、可选辅助说明（如「30 天内」「未删除」）、可选语义色调。禁止在指标条中放图表、迷你 sparkline、icon 圆形徽标、渐变背景、卡片阴影；指标条本身只有上下边线与列分隔线。
4. 语义色调（`tone`）仅在数值大于 0 时染色：
   - `warn`（暖色）用于「已停用」「即将到期」等需要关注但不紧急的状态；
   - `danger`（暖红）用于「已过期」「最后管理员保护触发数」等需要立即处理的状态；
   - 默认 `plain` 不染色，让正常计数保持中性。
5. 数值使用 `font-variant-numeric: tabular-nums` 等宽显示，多列数值上下对齐。
6. 窄屏（≤720px）允许水平滚动，不折行；折行会破坏对比阅读节奏。
7. 写操作后必须刷新指标条与列表（典型做法是 `Promise.all([loadList(), loadStats()])`）。指标条加载失败必须静默处理，不打扰用户的主要操作反馈。

### 总览页（ConsoleOverviewView）

控制台首页必须提供有意义的工作入口，不允许只有「欢迎语 + 角色卡片」。要求：

- 平台管理员视角：指标条（租户聚合） + 两列结构不同的实用列表（如「需关注 / 最近创建」），列表项可点击直接跳到下一步操作。
- 租户管理员视角：指标条（本租户成员聚合） + 一条快速入口；**不得伪造未实现的业务列表来填充空间**。
- 普通成员：保留极简欢迎语；本轮不实现额外业务。

### 禁止的设计反模式

控制台所有页面禁止出现以下模式（包括但不限于）：

- **大数字 + 渐变图标的指标卡墙**（impeccable 称为 hero-metric template）；用 MetricStrip 取代。
- **identical card grid**：同尺寸卡片堆 4-up / 6-up，每张只放 icon + 标题 + 一段说明。
- **side-stripe borders**：`border-left` 以彩色色块作为装饰；用全边框、背景轻色或不装饰取代。
- **gradient text**：标题或数值使用 `background-clip: text` 渐变。所有强调通过字重与字号实现。
- **小写大写字母 + 大间距的「眉签」（eyebrow）** 出现在每个区块标题之上；这是 2023-2024 年 SaaS 模板的批量味道。
- **`01 / 02 / 03` 风格的章节编号**作为常规视觉脚手架；只有真正承载顺序信息的流程才允许编号。
- **彩色玻璃拟态背景 / 大圆角 / 霓虹发光**；控制台默认 8-12px 圆角，禁止超出。
- **modal 作为第一选择**：能内联或渐进展开就不要弹窗；当前管理页的「管理面板」分段编辑已是标准范式，新增页面遵循。

### 留白、密度与色彩

- 主表格区与四周保持 24-28px 内边距；表格行内文字 13-14px，行高 1.5-1.6；列与列之间不靠竖线，靠空白节奏。
- 颜色使用 `styles.css` 中的 CSS 变量（`--bg`、`--surface`、`--surface-elevated`、`--text`、`--text-muted`、`--border`、`--accent`、`--danger`、`--focus-ring`），禁止页面内硬编码颜色（极少数语义点缀如 `#20a779` 成功绿、`#d98b20` 过期橙、`#b45309` 暖色除外，且必须在两种主题下都通过 4.5:1 对比度）。
- 状态展示统一使用「6-7px 圆点 + 文字」的内联标签，不使用彩色背景 Tag；密集表格里色块越多越乱。
- 可点击交互的 `:focus-visible` 必须使用全局 `outline: 2px solid var(--accent); outline-offset: 2px;`，禁止隐藏焦点环。

### 响应式

- 桌面 ≥1024px：完整布局，所有列可见。
- 平板 720-1023px：保持 desktop 骨架，可缩窄侧栏，必要时让指标条 / 工具栏弹性折行。
- 移动 ≤720px：侧栏抽屉化；指标条水平滚动；管理面板 modal 占满宽度；详情列表 `grid-template-columns` 从 100px 缩到 84px。
- 禁止使用「fluid clamp() heading」让正文随视口缩放——产品 UI 使用固定 rem 字号。

### 验收要求

新增或改造管理页必须在桌面 / 平板 / 移动三视口下手工或 Playwright 验证：

- 页面不出现横向溢出。
- 指标条数值在 0 / 中等 / 大数（>10000）三种规模下排版稳定。
- 主表格在 0 条、1 条、≥20 条三种规模下空状态、单行、分页都正确。
- 写操作成功后指标条数值与列表同步刷新，不需要刷新页面。
- 所有失败 toast / dialog 仍遵循「用户可理解错误提示规范」，不暴露 HTTP 状态码、错误编码、SQL、堆栈。

## 用户可理解的错误提示规范

所有页面和接口必须面向普通用户提供清晰、可理解、可操作的中文错误提示。

禁止将技术错误直接展示给用户。

### 禁止直接展示的内容

以下内容不得出现在页面、弹窗、Toast、表单提示或普通用户可见的错误页中：

```text
HTTP 状态码，例如 401、403、404、500
业务错误编码，例如 AUTH_INVALID_CREDENTIALS
Java 异常类名
异常堆栈
SQL 或数据库报错
数据库字段名
后端原始异常消息
内部路径、配置、Token、密码或其他敏感信息
```

## 后端接口规范

1. 接口仍必须返回正确的 HTTP 状态码，供前后端程序识别：

   * 登录凭证错误、未登录、登录失效：`401`
   * 无权限：`403`
   * 参数校验失败：`400`
   * 资源不存在：`404`
   * 服务内部异常：`500`

2. HTTP 状态码和业务错误码只用于程序逻辑，不得直接作为用户提示展示。

3. 后端应返回统一、安全、稳定的业务错误结构，至少包含：

   * 可供前端识别的业务错误编码
   * 用户可理解的中文提示
   * 可选的请求追踪标识

4. 后端不得返回异常堆栈、SQL、数据库错误、内部实现细节、密码哈希、Token 或配置内容。

5. 服务端应记录完整异常日志；普通用户只看到安全且友好的提示。

## 前端错误处理规范

前端必须建立统一错误处理和错误文案映射机制。

禁止直接展示后端原始 `message`、HTTP 状态码、业务编码或异常文本。

至少使用以下用户提示：

| 场景 | 用户可见提示 |
| --- | --- |
| 用户名或密码错误 | 用户名或密码错误，请重新输入 |
| 账号已停用 | 当前账号已停用，请联系管理员 |
| 所属租户已停用 | 所属租户已停用，暂时无法使用 |
| 所属租户已过期 | 所属租户已到期，请联系管理员续期 |
| 所属租户已删除 | 当前账号所属租户不可用，请联系管理员 |
| 登录状态失效 | 登录已失效，请重新登录 |
| 无页面或操作权限 | 当前账号没有此操作权限 |
| 表单字段不合法 | 在对应字段附近显示具体校验提示 |
| 租户码已存在 | 该租户码已被使用，请更换后重试 |
| 删除受保护自营租户 | 自营租户受保护，无法删除 |
| 停用受保护自营租户 | 自营租户受保护，无法停用 |
| 网络连接失败 | 网络连接失败，请检查网络后重试 |
| 请求超时 | 请求超时，请稍后重试 |
| 服务内部异常 | 服务暂时不可用，请稍后重试 |

## 交互要求

1. 登录失败必须显示“用户名或密码错误，请重新输入”，不得显示 `401 Unauthorized` 或类似技术文本。
2. 表单校验错误必须优先显示在对应字段附近，说明用户需要如何修改。
3. 非表单操作失败应使用统一 Toast、通知栏、错误页或反馈组件展示。
4. 提交失败后应保留用户已填写的内容，避免用户重复输入。
5. 登录失效时，提示“登录已失效，请重新登录”，再安全跳转至登录页。
6. 无权限访问页面时，展示友好的无权限提示或页面，并跳转到用户有权限访问的默认页面。
7. 开发环境可在浏览器控制台输出调试信息；生产环境不得向用户界面暴露技术细节。
8. 所有新增页面和接口必须复用统一错误处理机制，不得各自随意展示错误文本。

## 验收要求

使用 Playwright 或等价方式验证以下场景：

* 错误密码登录时，页面不显示 `401`、异常类名或后端原始错误。
* 无权限访问页面时，页面展示可理解提示，不显示 `403`。
* 表单校验失败时，字段附近显示明确中文提示。
* 网络失败、超时、服务异常时，用户看到可理解提示。
* 页面、Toast、弹窗和错误页中不出现 SQL、堆栈、业务编码或技术错误信息。
