# Fluxora Naive UI 全面视觉重构设计

## 目标与范围

本次重构覆盖 `fluxora-web` 中的公开首页、文档、登录、自营租户初始化、控制台概览和租户管理页面。目标是在不改变既有真实认证、权限、初始化和租户 CRUD 行为的前提下，建立一套由 Naive UI 驱动的统一界面系统。

界面应符合 Fluxora 的“克制、可靠、敏锐”定位：像工程工具而非传统后台模板或带有装饰性 AI 风格的营销页面。后端、接口契约、路由权限和数据流不在本次调整范围内。

## 设计方向

采用“Dark Ops”作为核心气质，而不是强制深色默认值：

- 首次访问跟随操作系统的亮暗偏好；用户可在所有公开和登录后页面切换主题，手动选择持久化保存。
- 亮色主题为冷白、浅灰和深墨色；暗色主题为近黑蓝灰和高对比浅灰。
- 冷蓝是唯一品牌色，只用于主操作、当前导航、键盘焦点和少量状态信号；成功、警告、错误使用独立的语义色。
- 字体采用苹果式系统栈：`SF Pro`、`PingFang SC`、`system-ui`。用户名、租户码、日期和技术字段使用等宽字体。
- 不使用暖黄底色、渐变背景、玻璃拟态、超大营销标题、无意义卡片墙或仅有图标而没有清晰含义的操作。

## 信息架构与页面布局

### 公开首页

首页使用轻量顶部导航和可在所有断点操作的主题切换。首屏包含一条简洁定位、两个明确入口和一个表现请求经由平台流转的结构化示意。后续按能力、接入流程和 FAQ 排列，使用排版、网格和分隔线建立层级，不制造伪造的产品数据或营销噪音。

### 文档

文档复用公开顶部导航。桌面端为左侧目录和限制行宽的正文，平板与移动端将目录切换为抽屉。当前章节、代码块、下一步链接和 FAQ 使用统一的阅读层次；目录折叠不影响主题切换和导航可达性。

### 登录

登录页使用居中的认证面板，而非传统左右分栏后台页面。品牌和主题切换始终可见；完整标签、字段说明、密码可见性切换、提交状态和字段级错误均由 Naive UI 表单承载。错误文本仅使用已有安全中文文案，提交失败时保留用户输入。

### 自营租户初始化

初始化页面是明确的两步流程：租户信息确认和首个租户管理员创建。进度条显示当前步骤；规则摘要始终显示租户码 `default`、自营属性和受保护限制。提交、成功和失败反馈复用统一表单与消息机制，刷新后仍依据真实后端状态恢复流程。

### 控制台

登录后所有一级菜单均位于左侧边栏。桌面端侧边栏和顶部栏固定，只有内容区滚动；顶部栏只展示当前位置、主题切换、用户菜单和全局操作。移动端将侧栏切换为 Naive UI 抽屉。概览页只显示可解释的欢迎、权限和待办状态，不用虚构指标填充卡片。

### 租户管理

页面顶部包含标题、说明和“新建租户”主动作。筛选工具栏支持展开、收起和一键重置。表格主列可进入详情；常用编辑操作为可见文字按钮，其余低频操作集中在有文字提示的“更多操作”菜单。详情使用标准 `NDrawer` 和 `NDrawerContent`，动作按一般、状态和危险三个区域排列；自营租户的限制以禁用态和原因文字同时说明。

## 前端架构

### 组件边界

新增且仅新增以下通用视觉组件：

- `PublicHeader`：公开首页和文档的导航、移动菜单与主题入口。
- `ThemeToggle`：带可读标签和无障碍名称的主题切换。
- `ConsoleShell`：侧栏、固定顶部栏、移动端抽屉和权限菜单。
- `PageHeader`：标题、说明、主动作和可选工具区域。
- `StatusTag`：租户类型、状态和受保护状态的统一语义标签。
- `AsyncState`：加载骨架、空状态、错误状态与重试动作。

每个业务页面只负责组装上述组件、Naive UI 组件和已有的服务调用。认证状态、权限判断、错误文案映射和租户数据接口继续使用现有 store 与 service，不在页面中复制实现。

### 主题与样式

主题 store 继续是唯一主题状态来源：无存储偏好时读取系统偏好，手动切换后持久化。Naive UI 的 `NConfigProvider` 通过一份严格类型化的 `themeOverrides` 获得全局颜色、排版、圆角、表格和反馈组件样式。

全局 CSS 只保留 reset、字体 token、颜色 token、焦点样式、基础布局和可访问性规则。以下遗留手写样式必须删除或由 Naive UI / 页面局部样式取代：`.button`、`.field`、`.login-form`、旧 `.console > aside`、`.console-main`、`.metrics`、旧 `.docs` 布局、手写抽屉、手写表格和无效的 CSS `aria-label` 声明。不得保留两套同时控制同一组件的样式规则。

## 响应式与可访问性

- 桌面：固定左侧控制台导航，表格完整展示，正文阅读宽度受限。
- 平板：侧栏压缩，操作区换行，表格可横向滚动并保持首要列可见。
- 手机：侧栏为抽屉，筛选默认折叠，表格隐藏低优先级列并通过详情查看完整信息。
- 所有交互组件均有文字或 `aria-label` 说明、可见焦点、禁用原因、键盘可达性和 150–250ms 的状态性过渡。
- 亮暗主题下的文字、边界、表单、状态标签和焦点都达到可读对比，不向普通用户展示技术错误、状态码或后端原始信息。

## 验证与完成标准

实施后必须执行：

1. `npm run build`，确保 TypeScript 与 Vite 生产构建通过。
2. 前端单元测试，确认路由、认证状态和租户操作交互未回退。
3. Playwright 真实联调：登录、初始化、租户 CRUD、权限菜单和无权限路由。
4. Playwright 多视口检查：桌面、平板、手机上的首页、文档、登录、初始化、控制台和租户管理；分别验证亮暗主题、移动导航、侧栏固定及菜单当前态。
5. 样式审计：全局样式中不保留被 Naive UI 替代的旧控件和布局规则；页面不再存在纯图标而无可读用途的核心操作。

## 非目标

- 不修改后端业务逻辑、数据库结构、接口返回、认证方案或权限模型。
- 不增加新的业务领域、Mock 数据、图表、营销动画或图片资产。
- 不替换 Naive UI，不引入第二套 UI 组件库，也不为视觉重构重写路由、请求层或状态管理。
